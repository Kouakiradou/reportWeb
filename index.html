Requirements

A progressive web-app to allow GOSH staff to design surveys for child patients, and for child patients to answer these surveys.

Gathering user requirements:

We’ve decided to carry out a semi-structured interview for the two users that would use our product, which are the Gosh Drive admins designing questionnaires, and the child patients answering them.


1) Gosh Drive Admin interview:
Q: Should these questionnaires have a fixed format?
A: No, it has to be able to be adjusted, like the colour and image could be changed as they wished.
Q: How do you think the data and answers from the questions should be stored?
A: A database could be used to store the answers from the questionnaires, as well as the login information of users designing the questionnaires.
Q: Should the web app keep track of the child patients answering the questionnaires?
A: It shouldn’t, instead, it could be anonymous to allow the kids to express themselves.

2) Child Patient Interview:
Q: Why didn’t you want to fill in the form?
A: The form is very boring.
Q: Do you like questionnaires to be colourful or plain?
A: Colourful.
Q: Would you like if we put some cute animals in the questionnaire?
A: Yes, I really like animals.



MosCow List of Requirements:

    • Admin panel for GOSH staff to create and manage multiple questionnaires.
    • Designed as a Progressive Web App to allow completion of questionnaires using tablets and phones.
    • Database to store answers from questionnaires.
    • Hosted on Heroku

    • Separate questionnaire designs for inpatients and outpatients.
    • Simple design and UI for patients answering questionnaires.
    • Acceptance of different types of answers for the questionnaire designs e.g. drop-down, MCQ, sliders, text.

    • Different UI designs for different patient age groups answering questionnaires.
    • Statistics to analyse the answers obtained from questionnaires e.g. trends, patterns, etc.

    • Log-in system for patients answering questionnaires.
    • Multilingual surveys.

Related Technologies:

As our application is a new full-stack application, we decided to choose separate frameworks for front-end development and back-end development. After doing spending time learning about these frameworks, we have narrowed down our choices:

  • Django

    Django is a Python framework which follows the model-template-view pattern and was recommended by our clients. In practice, we found it to have many built-in features such as a front-end administration panel, database mapping, and easy authentication [1]. However, time is required to get started to learn the basic Django boilerplate and structure.

  • Flask

    Flask is a light-weight Python framework that is easier to get started with. However, it offers much less built in functionality than Django, and requires developers to be flexible [1] to make the best design decisions. It is easy to develop a rest API in Flask, but functionalities such as authentication is much harder to be implemented in the future.

  • Vue

    • Open-source JavaScript progressive framework that follows the Model–view–viewmodel pattern.
    • Has a gentle learning curve [2] due to its simple API.
    • Fast and light-weight.
    • Heavy emphasis on Single File Components, allowing us to build style and functions within one file.

  • React

    • An extremely popular framework, hence resource availability is very high.
    • Requires the use of various other libraries for different functionality.
    • Has a slightly steeper learning curve [2] than Vue.
    • ‘create-react-app’ package helps us get started easily, and invokes service workers [3] that allows users to see results even when offline.

Back-end decision:

We have decided to use Django over Flask despite its steeper initial learning curve. This is because we believe Django lets us focus on the actual questionnaire designs in the long run, as a lot of functionality is already built-in. We also think that Flask may introduce many bugs in the future due to our inexperience in back-end development so far.



Front-end decision:

Both frameworks were reasonable to us, but we decided to choose React as our front-end JavaScript framework in the end. Despite having a steeper learning curve, we decided that the ‘create-react-app’ package can help us overcome this problem. The service workers also prove to be an advantage, as offline usage is an important property of a progressive web app.



Related Projects:

  • Survey Monkey

    Survey Monkey is a website designed particularly for users to design questionnaires. It is written in Python and developed with the .NET framework. It offers a wide range of functionalities, including functionalities to import questionnaires from templates, analyse results, and obtain verified responses for your questionnaires. It is very similar to our project, Hospital Surveys, as it allows for questionnaires that are highly customizable. However, not all of the functionality is to be implemented in our project.

  • Google Forms

    Google forms is also designed for users to design questionnaires, and for users to respond to these questionnaires easily and efficiently. Google Forms allows for users to design questionnaires to obtain responses in a short amount of time. In our project, we plan to implement the simple UI design for users to design questionnaires, but we require a more detailed way to process the data from the responses obtained, i.e. through a database.
